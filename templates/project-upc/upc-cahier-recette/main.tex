\subsection{Contexte et enjeux}

[Rappeler le contexte du projet et les enjeux de la phase de recette.]

Le cahier de recette est un document clé dans le cycle de développement d'un
logiciel. Il permet de vérifier que l'application développée est conforme aux
exigences définies dans le cahier des charges, avant sa livraison finale.

\subsection{Objectifs du cahier de recette}

Le cahier de recette vise à :
\begin{itemize}
  \item Vérifier la conformité de l'application avec le cahier des charges.
  \item Valider les fonctionnalités définies par l'encadrant et l'équipe.
  \item Identifier les éventuelles non-conformités avant la mise en production.
\end{itemize}

Ce document servira de référence pour l'ensemble des phases de validation,
depuis les tests unitaires jusqu'à la recette finale.

\subsection{Objectifs des tests}

[Définir les objectifs de la campagne de tests.]

\begin{itemize}
  \item Valider que chaque fonctionnalité fonctionne conformément aux
    spécifications.
  \item Vérifier la robustesse de l'application face aux entrées incorrectes.
  \item S'assurer que les performances répondent aux attentes.
  \item Tester la sécurité : authentification, autorisation, protection des données.
\end{itemize}

\subsection{Méthodes}

[Décrire les méthodes de test retenues.]

\begin{itemize}
  \item \textbf{Tests manuels :} exécution des scénarios par un testeur humain
    en suivant ce cahier de recette.
  \item \textbf{Tests automatisés :} utilisation de [framework de test] pour
    les tests unitaires et d'intégration.
  \item \textbf{Tests de non-régression :} exécution systématique de l'ensemble
    des tests après chaque modification majeure.
\end{itemize}

\section{Concepts de base}
% =============================================================================

[Définir les concepts clés à maîtriser pour réaliser les tests.]

\begin{description}
  \item[Cas de test] Scénario décrivant une situation à tester, avec les
    actions à réaliser et le résultat attendu.
  \item[Critère d'acceptation] Condition que l'application doit remplir pour
    qu'un test soit considéré comme réussi.
  \item[Régression] Apparition d'un défaut dans une fonctionnalité
    précédemment validée suite à une modification.
  \item[Recette] Phase finale de validation par le client/encadrant avant
    la mise en production.
\end{description}
